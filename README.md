# adama

> 基于 Go 构建的高并发秒杀微服务工程实践项目。

## 项目简介

`adama` 聚焦秒杀场景下的高并发下单问题，围绕库存预留、异步削峰、订单处理、事务协同、补偿恢复和服务治理，搭建了一套完整的后端微服务工程。项目采用 `Go + Kratos` 作为主框架，使用 `Redis` 承接热点库存路径，使用 `Kafka` 解耦异步下单链路，使用 `MySQL + Ent` 做持久化，使用 `etcd / Nacos` 做服务注册发现与配置管理。

这个项目不是单纯的 CRUD 服务，而是围绕以下核心问题展开：

- 高并发请求下如何避免超卖
- 如何把同步下单链路改造成异步削峰链路
- 如何在多服务协作下保证库存与订单状态尽量一致
- 如何把 gRPC、HTTP、Proto、配置、数据层和压测链路组织成统一工程骨架

## 技术选型

## 设计思路

### 1. 为什么使用 Redis

秒杀链路的核心瓶颈不是业务代码，而是库存更新带来的数据库热点竞争。项目将库存判断前移到 Redis，通过 `Lua` 完成库存预留、用户占位和重复请求拦截，优先保证热点路径的快速裁决能力。

### 2. 为什么接入 Kafka

下单、库存、订单落库等动作如果全部放在同步请求中，延迟和失败率都会显著升高。项目通过 Kafka 将秒杀请求与后续订单处理解耦，把请求主链路收缩为“快速校验 + 投递异步任务”，以此实现削峰和吞吐提升。

### 3. 为什么保留 Proto / gRPC / HTTP 双协议骨架

项目不仅提供 HTTP 接口，也保留 Proto 驱动的服务契约与代码生成能力。这样可以让请求结构、服务定义、HTTP 映射和生成代码保持一致，降低多服务协作时的接口漂移风险。

### 4. 为什么采用微服务拆分

项目将用户、商品、订单、任务、网关、运维问答拆分为独立服务，目的是把热点链路、异步链路、边缘流量治理和基础能力隔离开，便于后续分别演进、治理和压测，而不是把所有逻辑堆在一个单体应用里。

## 系统骨架

当前项目主要由 6 类核心服务组成：

- `app/goods/service`
  商品服务，负责商品信息、商品查询和部分事务协同入口

- `app/order/service`
  订单服务，负责下单、秒杀下单、库存扣减、异步消息投递

- `app/user/service`
  用户服务，负责用户注册、登录和用户信息维护

- `app/job/service`
  任务服务，当前已拆成 `task-worker`、`task-timeout`、`task-scheduler`，分别负责异步消费、超时关闭和修复巡检

- `app/gateway`
  网关服务，负责统一入口、路由转发、上下文透传和限流治理

- `app/ops-ai`
  运维问答服务，负责本地 RAG 检索、FAQ/文档查询和只读诊断能力

核心公共层包括：

- `api/`
  Proto 协议、请求响应结构、gRPC / HTTP 生成代码

- `pkg/`
  公共组件与工具函数，例如缓存 Key、加密、分页等

- `benchmarks/`
  wrk 压测脚本和压测结果记录

- `docs/`
  结构设计、Redis Key、数据库结构和问题记录

## 核心链路

秒杀下单的典型流程如下：

1. 请求进入订单服务
2. 通过 Redis Lua 完成库存预留、用户占位和重复请求拦截
3. 命中可下单条件后组装消息写入 Kafka
4. 由 `task-worker` 异步消费消息并初始化工作流
5. 在异步消费阶段执行 `DTM/TCC`，协调 `goods` 与 `order` 服务推进库存与订单状态
6. 结合工作流状态表、超时关闭、DLQ 和修复任务完成最终处理

## 性能数据

当前仓库内已记录多轮压测与优化数据，具代表性的结果包括：

- 单机下完整秒杀链路的稳定处理能力约 `240~290 单/秒`
- `gateway c128` 场景下，数据库索引和任务扫描查询优化后：
  - `P95` 从约 `798ms` 降至 `293ms`
  - `20s` 内异步完成率从约 `49%` 提升到 `99%`
- `GET /goods/list` 经 `gateway` 在高并发压测下仍可维持 `6000+ req/s` 级别吞吐
