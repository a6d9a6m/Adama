# adama

> 基于 Go 构建的高并发秒杀微服务工程实践项目。

## 项目简介

`adama` 聚焦秒杀场景下的高并发下单问题，围绕库存扣减、异步削峰、订单处理、服务治理和接口协议设计，搭建了一套完整的后端微服务骨架。项目采用 `Go + Kratos` 作为主框架，使用 `Redis` 承接热点库存路径，使用 `Kafka` 解耦异步下单链路，使用 `MySQL + Ent` 做持久化，使用 `etcd / Nacos` 做服务注册发现与配置管理。

这个项目不是单纯的 CRUD 服务，而是围绕以下核心问题展开：

- 高并发请求下如何避免超卖
- 如何把同步下单链路改造成异步削峰链路
- 如何在多服务协作下保证库存与订单状态尽量一致
- 如何把 gRPC、HTTP、Proto、配置、数据层和压测链路组织成统一工程骨架

## 设计思路

### 1. 为什么使用 Redis

秒杀链路的核心瓶颈不是业务代码，而是库存更新带来的数据库热点竞争。项目将库存判断前移到 Redis，通过 `库存预扣 + 售罄标记` 减少 MySQL 直接承压，优先保证热点路径的快速裁决能力。

### 2. 为什么接入 Kafka

下单、库存、订单落库等动作如果全部放在同步请求中，延迟和失败率都会显著升高。项目通过 Kafka 将秒杀请求与后续订单处理解耦，把请求主链路收缩为“快速校验 + 投递异步任务”，以此实现削峰和吞吐提升。

### 3. 为什么保留 Proto / gRPC / HTTP 双协议骨架

项目不仅提供 HTTP 接口，也保留 Proto 驱动的服务契约与代码生成能力。这样可以让请求结构、服务定义、HTTP 映射和生成代码保持一致，降低多服务协作时的接口漂移风险。

### 4. 为什么采用微服务拆分

项目将用户、商品、订单、任务拆分为独立服务，目的是把热点链路、异步链路和基础能力隔离开，便于后续分别演进、治理和压测，而不是把所有逻辑堆在一个单体应用里。

## 系统骨架

当前项目主要由 4 类核心服务组成：

- `app/goods/service`
  商品服务，负责商品信息、商品查询和部分事务协同入口

- `app/order/service`
  订单服务，负责下单、秒杀下单、库存扣减、异步消息投递

- `app/user/service`
  用户服务，负责用户注册、登录和用户信息维护

- `app/job/service`
  异步任务服务，负责消费消息队列并处理订单相关异步任务

核心公共层包括：

- `api/`
  Proto 协议、请求响应结构、gRPC / HTTP 生成代码

- `pkg/`
  公共组件与工具函数，例如缓存 Key、加密、分页等

- `benchmarks/`
  wrk 压测脚本和压测结果记录

- `docs/`
  结构设计、Redis Key、数据库结构和问题记录

## 核心链路

秒杀下单的典型流程如下：

1. 请求进入订单服务
2. 通过 Redis 判断库存与售罄状态
3. 命中可下单条件后执行库存预扣
4. 组装订单消息并写入 Kafka
5. 由异步任务服务消费消息并完成后续订单处理
6. 结合数据库持久化、事务协同和状态回写完成最终处理
