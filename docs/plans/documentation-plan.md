# 文档完善计划

> 目标：把当前仓库里的“项目介绍、架构说明、运行方式、链路设计、排障资料、测试与性能沉淀”整理成一套可读、可维护、可持续更新的文档体系。

## 背景

当前仓库已经具备较完整的后端工程能力，但文档仍然存在几个问题：

- 根目录说明、`docs/`、`scheduled.md` 和 AI 沉淀文档之间边界还不够清晰。
- 架构文档覆盖了部分主题，但缺少统一入口和阅读顺序。
- 关键链路已经实现，但“系统如何工作”与“出了问题怎么排查”还没有串起来。
- 一部分规划已经存在，但缺少一个单独面向“文档建设”的专项计划。

这份计划先解决“文档怎么组织、先补什么、怎么持续维护”的问题，后续再按任务清单逐项落地。

## 范围

本轮文档专项只处理文档资产，不涉及业务代码改造。

纳入范围：

- 根目录项目说明与文档索引
- `docs/architecture/` 下的系统与链路设计文档
- `docs/plans/` 下的规划文档
- `docs/ai/` 下的评审、测试、性能与知识沉淀
- `scheduled.md` 的执行任务清单
- FAQ、部署、压测、排障相关入口说明

暂不纳入范围：

- 直接补测试代码
- 直接修改服务实现
- 补完整 API 参考手册
- 引入新的文档站点生成器

## 目标产物

本轮完成后，文档体系至少要具备以下能力：

1. 新人能在 10 分钟内看懂项目结构、核心链路和本地运行方式。
2. 开发者能快速定位秒杀链路、任务补偿、网关暴露面和观测面文档。
3. 后续补测试、补监控、补排障记录时，有明确落点，不再把内容散落到根目录。
4. `scheduled.md` 只承担执行清单职责，`docs/plans/` 只承担规划职责。

## 文档结构约定

建议继续沿用当前目录，但强化职责边界：

- `README.md`
  项目总览、技术栈、核心能力、快速开始、文档入口。
- `docs/README.md`
  文档索引、阅读顺序、目录职责说明。
- `docs/plans/`
  面向未来的实施计划、专项计划、阶段任务说明。
- `docs/architecture/`
  面向当前实现的系统设计、链路设计、数据结构、事务与消息机制。
- `docs/ai/`
  AI 参与形成的评审、测试矩阵、知识库、性能分析与使用说明。
- `scheduled.md`
  当前正在执行和待执行的任务清单，强调顺序、状态、优先级。
- `FQA.md`
  高频问题、常见故障、标准排查入口。

## 分阶段计划

### 第一阶段：统一入口与阅读路径

目标：先解决“文档在哪看、先看什么”的问题。

输出：

- 补齐 `docs/README.md` 的阅读顺序
- 在 `README.md` 增加文档入口
- 明确根目录文档和 `docs/` 的职责边界

验收标准：

- 新人打开仓库后能知道先看哪个文档
- 不再出现“计划、排障、知识、任务”混放的问题

### 第二阶段：补齐核心架构文档

目标：把项目最有价值的后端能力讲清楚。

优先补齐的主题：

- 系统总览与服务边界
- 秒杀下单主链路
- 订单工作流与状态机
- Redis 预扣库存与幂等控制
- Kafka 异步削峰与失败补偿
- DTM / TCC / barrier 的使用边界
- gateway 暴露范围与限流策略
- task 角色拆分与定时任务职责

验收标准：

- 能从文档串起“请求进入到订单落库”的完整流程
- 能分清同步链路、异步链路和补偿链路

### 第三阶段：补齐运行与观测文档

目标：把“怎么跑起来、怎么观察、怎么排查”补完整。

优先补齐的主题：

- `docker compose` 本地运行说明
- 配置项说明与环境变量约定
- Prometheus / Grafana / exporter 说明
- 压测入口、基线、结果记录方式
- 常见异常的排查路径

验收标准：

- 能按文档完成本地启动、压测、观测和基本排障

### 第四阶段：补齐质量与沉淀文档

目标：让测试、评审、性能、问题复盘形成稳定沉淀。

优先补齐的主题：

- 测试矩阵与回归清单
- 风险评审模板
- 性能基线记录模板
- 事故 / RCA 模板

验收标准：

- 后续每次补测试、做评审、做压测都有固定落点

## 编写原则

- 先写“是什么、为什么、怎么用”，再写实现细节。
- 一份文档只解决一个核心问题，避免把计划、设计、排障混写。
- 用事实描述当前实现，用单独章节描述待完善项。
- 涉及状态流转、消息语义、补偿逻辑时，优先写清前置条件、状态变化和失败分支。
- 文档命名优先稳定、可检索，不追求花哨。

## 执行顺序

建议按下面顺序推进：

1. 调整文档索引与任务清单
2. 补系统总览与服务边界
3. 补秒杀主链路与订单工作流
4. 补网关、任务服务、消息与事务文档
5. 补运行、部署、观测与排障文档
6. 补测试矩阵、性能基线、评审模板

## 任务拆解原则

后续具体任务按以下规则拆：

- 单个任务只聚焦一个主题
- 每个任务都要有明确输出文件
- 优先补“高价值、跨模块、能减少沟通成本”的文档
- 先补当前代码已经稳定存在的能力，再补远期规划

## 完成定义

一个文档任务完成，至少满足：

- 已写入约定目录
- 已加入索引入口
- 已标明适用范围或对应模块
- 与当前代码实现不冲突
- 后续维护人能继续接着写
